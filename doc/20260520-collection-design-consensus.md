# 収集設計 合意事項（2026-05-20）

本書は、投資データ収集の運用合意を固定するための文書。

## 1. 基本方針

- DB-first を厳守する（判定・集計・通知はDB正本）。
- 生成ファイルは監査・互換・再処理用途に限定する。
- 収集は自動ジョブで回し、人手トリガー依存をなくす。
- サイト負荷に配慮し、取得間隔・上限・リトライを制御する。

## 2. データ粒度・優先度

- 価格データ粒度: `1d`（日足）固定。
- 情報ソース優先: `TDnet > Kabutan`。
- 信用可否が不明な銘柄: `保留`（採用しない）。
- ソース運用優先順（実装方針）:
  - 開示情報: `TDnet` を正本優先
  - 価格系列: `J-Quants` を第一候補、不可時は `yfinance` フォールバック
  - 補助材料: `Kabutan` / 手動観測（observations）

## 3. 鮮度・保持

- TDnet鮮度SLA: `当日中に取り込み完了`。
- Rawデータ保持期間: `14日`。
- 14日超のRawは定期クリーンアップ対象。

## 4. アラート閾値（初期値）

- coverage（当日対象銘柄で最低1件情報あり）
  - WARN: `< 70%`
  - ALERT: `< 50%`
- staleness（最終取得時刻）
  - WARN: `当日15:00時点で未更新`
  - ALERT: `当日終了時（23:59）でも未更新`
- new_rows_rate（新規/走査）
  - WARN: `< 5%` が3日連続
  - ALERT: `< 2%` が3日連続
- error_rate（収集リクエスト失敗率）
  - WARN: `> 10%`
  - ALERT: `> 20%`

## 5. 収集実行ガードレール

- リトライ上限: `3回`（指数バックオフ）。
- 日次収集上限:
  - TDnet: `1000件/日`
  - Kabutan: `200ページ/日`
- 保留銘柄の再判定頻度: `毎営業日1回`。

## 6. 実装対象（この合意に従う）

- 収集設計（Coverage駆動・未取得優先キュー）。
- テーブル統合集約（DBスキーマ整流化）。
- 既存スクリプト修正（DB-first化・重複排除・アラート整備）。

## 7. スキーマ方針（JSONハイブリッド）

- 固定列は最小限（検索・結合・集計に必須なキーのみ）。
- 可変項目は `payload_json` に格納して追従性を優先する。
- 新規項目はまず `payload_json` で受け、利用頻度が高い項目のみ固定列へ昇格する。
