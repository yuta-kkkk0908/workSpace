# Investment Priority Tasks

最終更新: 2026-05-19

## 目的

- 投資情報運用の精度と再現性を上げる。
- 母数不足によるルール誤昇格を防ぐ。
- 日次運用の判断を速くする。

## 優先タスク

1. 収集強化（母数拡大）
- `rough_backtest_full` の対象銘柄・イベントを拡張する。
- ルール昇格条件 `n>=20` を満たさない候補を自動で「昇格不可」に固定する。
- Kabutan系収集ジョブを分割し、失敗点を切り分ける。

2. バックテスト品質ゲート自動化
- 採用条件を固定する（`n>=20` / 期待値プラス / 最大DD閾値内 / 直近4週劣化なし）。
- 判定結果を `investment.db` のルール管理テーブルへ保存する。

3. シグナル生成の安定化
- `signals` 生成前に source 欠損チェックを追加する。
- 欠損時は「理由付き保留」を基本にし、無条件フォールバックを抑制する。
- `unknown` を出来高・ボラ・イベント種別で自動優先度付けする。

4. 朝/昼/夕ロジック最適化
- morning: 新規検知重視
- noon: 継続監視・失速検知
- evening: 翌日シナリオ向け再スコア
- 同一基盤ロジックを時間帯別重みで運用する。

5. 通知の意思決定向け再設計
- Signal通知を `新規候補 / 継続ウォッチ / 除外(理由)` の3区分で出す。
- 各候補に `rule_id / n / EV` を1行添付する。

6. 監視KPIの固定化
- 日次で `signals生成件数 / rule適用率 / unknown残数 / paper反映率 / n<20ルール比率` を出力し、劣化を早期検知する。

## 完了条件（Definition of Done）

- `n>=20` 未満ルールが自動で昇格不可になる。
- ルール採用判定が `investment.db` 上で再現可能になる。
- Signal通知で判断理由（`rule_id / n / EV`）が確認できる。
- KPIが日次で自動出力される。
